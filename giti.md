# Команды по работе с Git

### 1. Проверка версей git
```sh
git --versions
```
### 2. Глобальная настройка пользователя для всех проектов

```sh
git config --global user.name "Имя пользователя"
git config --global user.email "Почта пользователя"
```

### 3. Проверка имени и электронной почты:

```sh
git config user.name
git config user.email
```

### 4. Проверка глабальных настроек

```sh
git config --global --list
```
### 5. Проверка настроек текущего репозитория

```sh
git config --list
```

### 6. Инициализация Git репозитория в текущей дериктории:

```sh
git init
```

### 7. Задать другое имя для создания ветки по умолчанию

```sh
git config --global init.defaultBranch main
```

### 8. Проверка состояния:

```sh
git status
```
### 9. Посмотреть связь с удалённым репозиторием (GitHub)

```sh
git remote -v
```
### 10. Добавить содержимое файла в индекс:

```sh
git add <Имя файла>
```
### 11. Записать изменения в репозиторий:

```sh
git commit -m "коментарий"
```
### 12. Типы коммитов (по содержанию)
12.1 По структуре изменений
* Атомарные коммиты - одна логическая feature/fix
* Структурные коммиты - рефакторинг без изменения функциональности
* Функциональные коммиты - добавление новой функциональности
* Исправления - фиксы багов

12.2 По соглашениям (Conventional Commits)
```sh
feat: добавить аутентификацию пользователя
fix: исправить утечку памяти в модуле X
docs: обновить документацию API
style: форматирование кода (без семантических изменений)
refactor: переписать модуль обработки данных
test: добавить тесты для компонента Y
chore: обновить зависимости, настройки сборки
```
### 13. Структура хорошего коммита:
```sh
Краткое описание (до 50 символов)

Подробное описание (если нужно):
- Что было изменено
- Почему это было изменено
- Какие побочные эффекты

Fixes #123  # ссылка на issue
```
### 14 Пример хорошего коммита:
```sh
feat: добавить валидацию email при регистрации

- Добавлена проверка формата email на стороне клиента
- Реализована проверка уникальности email на сервере
- Добавлены соответствующие тесты

Closes #45
```
### 15 Первый коммит после инициализации
Классический первый коммит:
```sh
git init
git add .
git commit -m "Initial commit"
```
Более описательные варианты:
```sh
git commit -m "chore: initialize repository"
git commit -m "Initial project setup"
git commit -m "feat: add base project structure"
```
Что обычно включают в первый коммит:

* README.md с описанием проекта
* .gitignore файл
* Базовая структура проекта
* Лицензия (если есть)
* Конфигурационные файлы

### 16. Отмена изменений в конкретном файле:
```sh
git checkout -- src/Main.java # путь указан в качестве примера
```
### 17. Отменить все незакоммиченые файлы:
```sh
git restore .
# или
git checkout -- .
```

### 18. Посмотреть историю коммитов

```sh
git log --oneline
git log --oneline -5 # последние пять коммита
```
### 19. Графическое представление истории коммитов

```sh
git log --online --graph --all # компактный график с однострочными коммитами
```

```sh
git log --graph --all # Подробный график
```

### 20. Переход по версиям:

```sh
git checkout <четыре первых символа>
```

### 21. Возврат к актуальной версии

```sh
git ceckout main
```

## Работа с ветками

### 1. Создание и переключение веток

1.1 Создать новую ветку

```sh
  git branch <имя-ветки>
```
1.2 Переключиться на ветку

```sh
git checkout <имя-ветки>
```
1.3 Создать и сразу переключиться (комбинация)

```sh
git checkout -b  <имя-ветки>
git switch -c <имя-ветки> # в современных версиях Git
```
### 2. Просмотр веток

2.1 Список локальных веток
```sh
git branch
```
2.2 Список всех веток (локальных и удаленных)
```sh
git branch -a
```
2.3 Список веток с последним коммитом
```sh
git branch -v
```
2.4 Показать связи с удаленными ветками
```sh
git branch -vv
```
### 3. Удаление веток

3.1 Удалить локальную ветку (только если слияние завершено)
```sh
git branch -d <имя-ветки>
```
3.2 Принудительное удаление ветки
```sh
git branch -D <имя-ветки>
```
3.3 Удалить удаленную ветку
```sh
git push origin --delete <имя-ветки>
```
### Сценарии работы с ветками
### Сценарий 1: Разработка нового функционала
```sh
# Начинаем с основной ветки
git checkout main

# Создаём ветку для нового функционала
git checkout -b <имя-ветки>

# Работаем, делаем коммиты...
git add .
git commit -m "add new functional"

# Когда функционал готов - сливаем с main
git checkout main
git mtrge <имя-ветки>
```
### Сценарий 2: Исправление бага
```sh
# Создаём ветку для исправления
git checkout -b fix-bug

# Исправляем баг, коммотим
git add .
git commit -m "Fix bug"

# Возвращаемся в main для слияния
git checkout main
git merge fix-bug
```

# Работа с GitHub
1. Клонирует репозиторий с GitHub:

```sh
git clone <скопированная ссылка с GitHub>
```

Порядок действий:

* Находим репозиторий на GitHub
* Переходим во вкладку "code"
* Копируем ссылку

2. Создание репозитория на GitHub:

Порядок действий:
* Создать аккаунт на GitHub
* Нажимаем плюсиk в верхнем правом углу
* Выбераем "New repository"
* Присвае присваиваем имя для репозитория
* Копируем подсказки с GitHub и вставляем в локальный репозиторий

3. Направить локальный  репозиторий с изменениями на GitHub:

Порядок действий:

* Создать локальный репозиторий
* Сделать хотябы один "commit"
* Подружить локальный и удалённый репозиторий. GitHub при создании нового репозитория подскажет как это можно сделать
* Отправить "push" локальный репозиторий на GitHub, при этом необходимо авторизоваться

```sh
git push origine main
```

4. Принимает актуальную версия с GitHub:

Порядок действий:
* Внести изменения на GitHub
* Выкачать "pull" актуальную версию с удалённого репозитория

```sh
git pull origine main
```
5. Внесение собственного вклада в существующий репозиторий __pull request__
Порядок действий
* Делаем "__fork__" отдельная копия интересующего репозитория
* Делаем "__git clone"__ для нашей версии репозитория
* Создаём ветку с предлогаемыми изменениями
* Производим изменения только в этой ветке
* Отправляем эти изменения на свой аккаунт "__push"__
* В окне на GitHub появляется возможность отправить "__pull request__"

# Создание token доступа к персональному доступу GitHub.
Порядок дейсьвий:
* В правом верхнем углу любой страницы нажмите на фотографию своего профиля, затем нажмите __Settings__
* В левой боковой панели нажмите __Developer settings__
* В левой боковой панели нажмите Personal __access tokens__
* Нажмите Generate new __token__
* Даём token имя (любое имя описывающее цель его создания)
* Выберите области действия или разрешения, которые вы хотите предоставить этому token. Для того, чтобы использовать ваш __token__ для доступа к репозиториям из командной строки, выберите __repo__
* Нажмите __Generate token__
* копируем token в буфер обмена. По соображениям безопасности, после ухода со страницы, снова увидеть token не получится
